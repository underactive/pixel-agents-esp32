# Audit: Idle Screensaver — Characters Do Random Activities

## Files Changed

- `firmware/src/config.h`
- `firmware/src/office_state.h`
- `firmware/src/office_state.cpp`
- `firmware/src/renderer.cpp`
- `firmware/src/protocol.cpp` (immediate dependent)

---

## 1. QA Audit

**[FIXED] Q4 (HIGH):** Same as S4 — `startWalk()` failure leaves `idleActivity` set but character stuck in IDLE with stale activity state. Fixed by checking walk success after `startWalk()`.

**[FIXED] Q1 (MEDIUM):** Same as S6 — facing direction set before `startWalk()` is overwritten during walk; characters face wrong direction at interaction points. Fixed by storing/restoring direction via `activityDir` field.

**[FIXED] Q3 (MEDIUM):** Same as D11 — `getFrameIndex()` ACTIVITY case and `drawCharacter()` have fragile coupling for frame selection. Addressed with `// WHY:` comment documenting the dependency.

**Q8 (MEDIUM):** More frequent BFS calls from activity feature increase stack pressure (~1400 bytes on CYD, ~1000 on LILYGO per call). → Accepted: pre-existing pattern, well within 8KB task stack. Worth monitoring if more features pile on.

**Q5 (LOW):** Socializing direction computed toward target's current tile, not destination. Target may move before arrival. → Accepted per plan (Risks §3).

**Q2 (LOW):** `hitTestCharacter` sitting offset doesn't handle ACTIVITY, but ACTIVITY characters never sit. Correct as-is.

**Q9 (LOW):** `wanderCount` grows unboundedly (uint8_t wraps). Pre-existing, not introduced by this feature.

**Q10 (LOW):** Interaction points are within LILYGO's 10-row grid. Pre-existing furniture at rows 12-13 renders off-screen on LILYGO but is unrelated to this feature.

---

## 2. Security Audit

**[FIXED] S4 (HIGH):** `pickActivityTarget()` sets `ch.idleActivity` before `startWalk()`, but `startWalk()` can fail (no BFS path). On failure, character remains IDLE with orphaned `idleActivity` set, causing incorrect ACTIVITY transition on next successful walk.

**[FIXED] S6 (MEDIUM):** Facing direction set in `pickActivityTarget()` via `ch.dir` is overwritten during walk movement updates (lines 623-626). By the time ACTIVITY state is entered, the character faces the last walk direction, not the intended activity direction.

---

## 3. Interface Contract Audit

**[FIXED] IC-1 (HIGH):** Protocol validation in `protocol.cpp` line 121 uses `CharState::DESPAWN` (6) as upper bound, but `ACTIVITY` (7) exceeds it. This is intentionally correct — ACTIVITY is firmware-only and never sent over the wire — but lacked a clarifying comment.

**IC-2 (MEDIUM):** Companion bridge `pixel_agents_bridge.py` has no `STATE_ACTIVITY = 7` constant. → Accepted: companion never sends ACTIVITY; it only sends OFFLINE/TYPE/READ/IDLE. No code path references it.

---

## 4. State Management Audit

**[FIXED] S3 (HIGH):** Same as S4 — `idleActivity` mutation before `startWalk()` can leave orphaned state on walk failure.

**S5 (LOW):** `activityCooldown` is a boolean, not a timer. A character that completes one normal wander immediately after an activity can start a new activity. The plan specifies `ACTIVITY_COOLDOWN_SEC = 3.0` but this constant is unused; cooldown is implemented as "force one normal wander" via boolean flag. → Accepted: plan says "force normal wander before next activity", which the boolean achieves.

---

## 5. Resource & Concurrency Audit

**[FIXED] R3 (MEDIUM):** Same as S6 — facing direction lost during walk, confirmed independently.

No other resource, concurrency, or timing issues found. New fields add 6 bytes per character (36 bytes total), negligible.

## 6. Testing Coverage Audit

**T1 (MEDIUM):** No automated tests for activity state transitions, interaction point selection, or agent preemption of activities. → Accepted (no test harness in project).

**T2 (LOW):** Manual testing checklist items needed for: activity animations, cross-zone movement, agent preemption during activity, cooldown behavior. → Added to testing-checklist.md.

## 7. DX & Maintainability Audit

**[FIXED] D11 (LOW):** ACTIVITY frame selection split between `drawCharacter()` (standing pose override) and `getFrameIndex()` (READ frames for READING). Added `// WHY:` comment explaining the split.

**D6 (LOW):** `pickActivityTarget()` is 76 lines. → Accepted: logic is linear with clear branches per activity type. Extracting sub-functions would fragment without reducing complexity.

**D3 (LOW):** `InteractionPoint` tables in `config.h` use hardcoded tile coordinates. → Accepted per project conventions (all layout data is hardcoded, generated by layout editor).
